Fecha de corte: 2026-03-22

1. OBJETIVO GENERAL

El sistema fue disenado para gestionar anomalias, acciones, tareas, notificaciones y trazabilidad operativa dentro de un entorno corporativo. La arquitectura actual ya permite operar localmente en planta y, al mismo tiempo, deja preparada la migracion futura a infraestructura cloud sin cambios fuertes de arquitectura.

2. ARQUITECTURA GENERAL

Backend:
- Django
- Django REST Framework
- PostgreSQL
- JWT para autenticacion

Frontend:
- React
- TypeScript
- Vite
- PWA base

Despliegue:
- Docker Compose local
- Nginx para servir frontend y hacer proxy de API
- Gunicorn para backend
- Volumenes persistentes para datos, adjuntos, logs y backups

Estados principales:
- registered
- in_evaluation
- in_analysis
- in_treatment
- pending_verification
- closed
- cancelled
- reopened

Reglas ya implementadas:
- Toda anomalia tiene ID unico
- Toda transicion relevante se audita
- No se puede cerrar sin verificacion minima
- No se puede completar aprendizaje sin registrar learning
- Si falta informacion obligatoria, el backend bloquea la transicion
- La logica critica del workflow vive en backend

Cuando todas las tareas de un tratamiento estén en estado COMPLETADAS, el sistema debe cambiar automáticamente el 
estado del tratamiento a "VERIFICACION_DE_EFICACIA".

En este estado, el ADMIN debe:
- Cargar fecha de verificación
- Asignar responsable verificador

El verificador debe:
- Cargar evidencia objetiva (archivos)
- Indicar resultado: EFICAZ o NO EFICAZ
- Cargar comentarios
- Cargar fecha de verificación

Si el resultado es EFICAZ:
- El ADMIN puede cerrar el tratamiento -> estado CERRADO_EFICAZ

Si el resultado es NO EFICAZ:
- El ADMIN debe crear un NUEVO TRATAMIENTO
- El nuevo tratamiento debe estar asociado al tratamiento original (tratamiento_origen_id)
- El tratamiento original queda en estado CERRADO_NO_EFICAZ

Debe existir historial de cambios de estado y trazabilidad entre tratamientos.
No se puede cerrar un tratamiento sin verificación de eficacia.
